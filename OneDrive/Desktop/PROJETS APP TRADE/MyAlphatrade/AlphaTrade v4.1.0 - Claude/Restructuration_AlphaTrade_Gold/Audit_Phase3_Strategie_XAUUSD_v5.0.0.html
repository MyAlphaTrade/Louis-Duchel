Phase 3 — Audit

Audit de la stratégie XAUUSD actuelle + extensibilité

Analyse de code (pas de modification) : logique de décision réelle, fonctionnement effectif des 4 modes de stratégie existants, et étude d'architecture pour permettre l'ajout facile de nouvelles stratégies — notamment la future stratégie de gestion de position portée de KB1000.

1. Comment la décision d'entrée est calculée aujourd'hui

Tout part de compute_signal() dans alphatrade_engine.py. Le score BUY/SELL (0-100) est construit par addition pondérée de plusieurs composantes :

Le signal n'est retenu que si la confiance dépasse confidence_min (ajusté par un offset appris). Un second filtre — le gouverneur quantitatif (quant_veto) — peut bloquer l'entrée même si le signal est valide, en cas de risque de régime élevé (écart BUY/SELL trop faible, conflit multi-timeframe, sur-réaction de la bougie rapide, retournement de bougie).

Fonctionnel

Ce n'est pas un habillage : chaque composante est réellement calculée à partir de données de marché réelles (mt5.copy_rates_from_pos) et pèse effectivement dans le score final. Le "Consensus quantitatif" affiché au Tableau de bord reflète ce calcul réel.

2. Les 4 modes de stratégie sont-ils réellement fonctionnels ?

Oui, en partie — avec un vrai bug de désynchronisation trouvé. Voici comment ça devrait marcher, puis ce qui casse en pratique.

2.1 — Ce qui fonctionne réellement

Le mode choisi (scalping_fast / scalping_safe / long_analysis / combined) influence deux choses côté moteur Python :

  • Le poids du biais multi-timeframe dans le score (ligne 839-847) : 18 en Analyse longue, 12 en Mode combiné, 10 en Scalping prudent, 8 en Scalping rapide.
  • Le blocage d'entrée en cas de conflit de tendance large (ligne 2885-2896) : en combined/long_analysis uniquement, une entrée est refusée si la tendance multi-timeframe contredit le signal — sauf retournement confirmé.
  • effective_params_for_strategy() (ligne 266) applique aussi, par profil, un timeframe/confidence_min/cadence_sec/profit_target/max_hold_sec différent par défaut — mais seulement si l'utilisateur n'a jamais modifié/sauvegardé ce champ.

2.2 — Le bug trouvé : trois définitions du même mode, non synchronisées

Bug confirmé

Le mode de stratégie est défini trois fois indépendamment, avec des valeurs différentes :

EmplacementRôleExemple — scalping_fast
STRATEGY_PROFILES (Python)Profil réellement utilisé par le moteur à chaque décisionconfidence_min: 55, cadence_sec: 10, max_hold_sec: 600
strategyProfiles (renderer.js)Valeurs écrites dans le formulaire + params au clic sur le sélecteurconfidence_min: 64, cadence_sec: 15, max_hold_sec: 90
<option> (index.html)Liste affichée dans le sélecteur (juste des libellés, sans valeurs)—

Conséquence concrète : quand tu choisis un mode dans le sélecteur, applyStrategyProfile() (renderer.js ligne 1725) écrit ses propres valeurs dans les champs du formulaire et dans params. Si tu cliques ensuite "Sauvegarder", ces valeurs sont écrites dans params.json comme des valeurs "choisies par l'utilisateur". Résultat : côté Python, effective_params_for_strategy() (ligne 282) compare la valeur sauvegardée au défaut d'origine — comme elle a changé, le moteur considère que tu as personnalisé ce champ, et n'appliquera plus jamais sa propre définition du profil pour ce paramètre, même si tu reviens plus tard sur ce même mode. Le mode affiché et le comportement réel du moteur divergent silencieusement dès la première sauvegarde.

2.3 — Reste mort dans le code

Résidu mineur

Ligne 846 : elif strategy_mode in ("scalping_safe", "synthetic_scalp"): — synthetic_scalp a été retiré de STRATEGY_PROFILES et du sélecteur UI lors de l'isolation Or, mais cette référence traîne encore. Inoffensif (mode inaccessible désormais) mais à nettoyer.

3. Où se trouve la gestion de position (sorties) aujourd'hui ?

Recherche exhaustive : strategy_mode n'apparaît jamais dans la logique de sortie (trailing, break-even, rebond, clôtures partielles). Ces mécanismes sont globaux et uniformes, appliqués de la même façon quel que soit le mode de stratégie choisi.

Sélecteur "Mode de stratégie"
→
Décision d'ENTRÉE seulement
(timeframe, confiance, poids MTF, blocage conflit)
Aucun sélecteur
→
Gestion de POSITION (identique pour tous les modes)
(trailing, break-even, rebond, clôtures partielles)
Point d'architecture important

Le "Mode de stratégie" actuel est donc exclusivement un profil de style d'entrée, pas une stratégie complète au sens où tu l'entends pour KB1000. La stratégie KB1000 que tu veux porter est avant tout une stratégie de gestion de position (break-even réel, paliers de take-profit, clôtures partielles) — un axe aujourd'hui totalement séparé et non pilotable par mode. L'ajouter "dans" le même sélecteur suppose donc d'élargir ce que "mode de stratégie" veut dire : ne plus être seulement un profil d'entrée, mais un couple (profil d'entrée + comportement de gestion de position).

4. Proposition d'architecture — registre de stratégies extensible

Objectif : qu'ajouter une stratégie (KB1000 ou une future) ne demande plus qu'une seule déclaration, au lieu de toucher 3 fichiers en espérant qu'ils restent synchronisés (cause du bug ci-dessus).

4.1 — Source unique de vérité côté Python

Étendre STRATEGY_PROFILES pour que chaque entrée décrive, en plus du profil d'entrée déjà existant, sa gestion de position :

STRATEGY_PROFILES = {
  "scalping_fast": {
    "label": "Scalping rapide",
    "description": "...",
    "global": {...}, "symbols": {...},        # existant, inchangé
    "position_management": "generic"           # nouveau : handler de sortie utilisé
  },
  ...
  "kb1000_position": {
    "label": "Gestion de position KB1000",
    "description": "Break-even réel calibré broker, paliers de TP, clôtures partielles natives MT5.",
    "global": {...}, "symbols": {...},
    "position_management": "kb1000"             # nouveau handler, code séparé
  },
}

POSITION_MANAGERS = {
  "generic": manage_position_generic,           # logique de sortie actuelle, renommée/isolée
  "kb1000":  manage_position_kb1000,             # nouveau module, Phase 9
}

Dans la boucle de gestion des positions ouvertes, un seul point de branchement :

handler = POSITION_MANAGERS.get(profile.get("position_management", "generic"))
close_reason = handler(position, params, ...)

4.2 — Le formulaire génère la liste, il ne la recopie plus

Aujourd'hui index.html a une liste <option> écrite en dur, et renderer.js a sa propre copie strategyProfiles avec des valeurs différentes (la source du bug §2.2). La correction : le backend expose la liste des stratégies disponibles (labels + valeurs) via l'IPC déjà utilisé pour charger les paramètres, et le frontend construit dynamiquement le sélecteur et les valeurs appliquées à partir de cette unique réponse — plus aucune valeur numérique dupliquée côté JS.

4.3 — Ce que ça change pour l'ajout de KB1000 (Phase 9)

  • Écrire manage_position_kb1000() (break-even calibré broker via tick_value/tick_size/spread/lot, paliers de TP configurables, clôtures partielles natives MT5) — un seul nouveau module Python, sans toucher au reste du moteur.
  • Ajouter une entrée dans STRATEGY_PROFILES avec "position_management": "kb1000".
  • Le sélecteur "Mode de stratégie" dans Paramètres l'affiche automatiquement (généré dynamiquement, §4.2) — aucune modif HTML/JS supplémentaire.
  • Une fois sélectionné et sauvegardé, l'application l'exécute exactement comme les autres modes : le moteur route vers manage_position_kb1000 pour la gestion de position, tout en gardant le profil d'entrée (timeframe/confiance/cadence) que ce mode définit.
Réponse directe à la demande

Avec cette architecture, ajouter une stratégie future ne nécessite plus de savoir jongler entre 3 fichiers : une déclaration dans STRATEGY_PROFILES + un handler de gestion de position suffisent, et le sélecteur de Paramètres se met à jour tout seul.

5. Ce que cette phase ne fait pas (rappel méthode)

Conformément au cycle en 8 étapes, cette Phase 3 est un audit + une proposition d'architecture — aucun code n'a été modifié. Le bug de désynchronisation (§2.2) sera corrigé lors du codage réel de cette refonte (probablement rattaché à la Phase 4/5, puisqu'il touche directement le mécanisme que le nouveau AI Brain devra piloter), pas avant ta validation.
