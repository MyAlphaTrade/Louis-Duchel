Paramètres — Sur près de 110 paramètres recensés, 19 sont des leurres : l'utilisateur peut les modifier et sauvegarder dans l'app, mais le moteur ne les lit jamais (ex. tp_pips, max_position_loss, max_floating_loss, tout le bloc "Machine à états Renfort/Rebond"). 4. Assistant IA — Bonne nouvelle : Anthropic (Claude) est déjà intégré et prioritaire sur OpenAI pour le chat, avec les deux clés API déjà configurées. Pour les décisions de trade, un mode double-validation (Anthropic + OpenAI, consensus requis) est utilisé. Retirer OpenAI est un nettoyage ciblé sur 2 fichiers, pas une intégration à construire.
5. Origine "AVA" — Confirmé : "AVA / IA EXTERNE" est un libellé hardcodé en dur côté JavaScript pour toute origine classée EXTERNAL_AI côté moteur. Le vrai commentaire MT5 du robot (qui pourrait révéler "KB1000") est lu par le moteur puis jeté — jamais stocké, jamais transmis à l'affichage. Correction possible sans refonte lourde.

3Stratégie réelle & Microstructure / Sessions IA

Quelle stratégie est réellement implémentée, et qu'est-ce qui, dans les onglets Microstructure et Sessions IA, influence vraiment les trades ?

3.1 — Nature de la stratégie XAUUSD

Un système de règles techniques classiques avec pondération apprise de façon incrémentale — pas un modèle IA entraîné. Indicateurs bruts sur bougies M5 : EMA9/21/50, RSI14, MACD (EMA12−EMA26), momentum sur 5 bougies, position dans la zone haute/basse ("edge"). Chaque composant (trend, rsi, macd, edge, momentum) reçoit un score +1/0/-1, pondéré par un poids appris borné [0.65, 1.35]. Score BUY/SELL final = base 25 + bonus pondérés + biais multi-timeframe, clampé 0-100 (symbol_analysis, engine.py:939-1099).

Le mécanisme "d'apprentissage" affiché dans Sessions IA

C'est exactement ce qui produit les textes vus à l'écran ("le composant le plus renforcé est macd", "seuil ajusté de -2.20 points") : à chaque position fermée, reward = profit / max(MFE, MAE, 0.25), et pour chaque composant aligné avec la direction du trade, weights[name] += 0.025 × reward × alignment (clampé). Le seuil de confiance dérive de confidence_offset : +0.45 si perte, -0.12 si gain (clampé [-4, 10]). Ce n'est pas un entraînement ML classique (pas de descente de gradient, pas de fonction de perte globale) — c'est un renforcement heuristique incrémental de poids appliqués à des règles déjà câblées en dur (track_position_contexts, engine.py:1641-1745).

Cascade de blocages avant l'ouverture d'un trade

Un signal candidat traverse successivement : protection session/jour → sécurité lot → zone extrême sans confirmation → RSI → signal M5 rapide contradictoire → écart BUY/SELL minimum → biais multi-timeframe opposé → gouverneur de risque quantitatif (quant_veto, bloque si regime_risk ≥ 85 — règles fixes, pas d'IA) → confirmation serveur IA. Seul un signal ayant passé tous ces filtres ouvre réellement une position (auto_trade_step, engine.py:4040-4188).

Logique de sortie

  • Trail dynamique par paliers : sortie PROFIT_TRAIL selon le pic de profit atteint (4 paliers configurables et réellement actifs).
  • Momentum exit : sortie si le score opposé dépasse momentum_exit_score alors que la position est déjà en profit minimal.
  • Correction factuelle le trail dynamique/momentum exit est une fonctionnalité de la Phase 2.2 (absente du backup Phase 2.1), pas de la Phase 2.1 comme le nom pourrait le suggérer.

3.2 — Module Microstructure : confirmé 100% décoratif

Le module calcule bien OBI (order book imbalance), OFI (order flow imbalance), Kyle λ et POC (point de contrôle du volume) — code réel et correct dans market_microstructure.py. La classe se déclare elle-même "Moteur d'observation sans aucune méthode d'exécution d'ordre", renvoie "mode": "OBSERVATION_ONLY", "execution_authorized": False.

Vérification exhaustive dans alphatrade_engine.py : les valeurs OBI/OFI/Kyle λ/POC sont injectées dans status.json (payload pour l'UI) mais ne sont lues par aucune fonction de décision (symbol_analysis, auto_trade_step, position_exit_reason). Confirmé aussi côté UI : le texte affiché dit explicitement qu'il s'agit d'un module d'observation uniquement.

5Origine des trades externes — "AVA" hardcodé

Pourquoi tous les trades détectés comme IA externe s'affichent "AVA / IA EXTERNE", même pour des robots comme KB1000 ?

5.1 — Logique de détection côté moteur

# engine.py:23-24 MAGIC = 20260607 # magic number du bot AlphaTrade lui-même AVA_MAGIC = 7525001 # magic number d'UN SEUL robot externe nommé "AVA" # engine.py:1122-1134 — trade_origin() si magic == MAGIC ou comment contient "alphatrade"/"alphakaris" → "BOT" si magic == AVA_MAGIC ou comment contient "ava"/"bridge" → "EXTERNAL_AI" si magic != 0 (tout le reste, y compris KB1000) → "EXTERNAL_AI" sinon → "MANUAL"
Le système est binaire à 3 seaux (BOT / EXTERNAL_AI / MANUAL) — aucune table de mapping magic_number → nom_app n'existe. Python renvoie correctement un seau générique "EXTERNAL_AI" pour n'importe quel robot externe inconnu (donc KB1000 y tombe aussi, à raison). Le vrai commentaire MT5 de l'ordre — qui pourrait révéler "KB1000" — est bien lu par trade_origin() pour tester les mots-clés, puis immédiatement jeté : la table SQLite trades (engine.py:486-500) n'a même pas de colonne comment.

5.2 — Où "AVA" est réellement câblé en dur

// renderer.js:995-1000 — originLabel() function originLabel(origin) { if (key === 'BOT') return 'ALPHATRADE'; if (key === 'EXTERNAL_AI') return 'AVA / IA EXTERNE'; ← câblé en dur ici return 'MANUEL'; }

C'est exactement cette ligne côté JavaScript qui traduit systématiquement EXTERNAL_AI par "AVA / IA EXTERNE", quel que soit le robot réellement détecté. La fonction ne reçoit que la catégorie (BOT/EXTERNAL_AI/MANUAL), jamais le commentaire MT5 brut — qui n'existe de toute façon plus à ce stade côté frontend.

5.3 — Chaîne causale complète

  1. Un robot externe (ex. KB1000 sur BOOM1000) ouvre un ordre MT5 avec son propre magic et un comment qui pourrait contenir "KB1000".
  2. trade_origin() ne reconnaît ni le magic ni le comment → classe correctement en "EXTERNAL_AI" (comportement backend correct, c'est un seau générique).
  3. Le comment original est perdu — pas de colonne DB, pas de champ transmis pour les trades fermés.
  4. originLabel() traduit systématiquement EXTERNAL_AI par le libellé fixe "AVA / IA EXTERNE", quel que soit le robot réel.
"AVA" n'est donc ni un vrai mapping, ni une capacité technique volontairement écrasée — c'est le libellé d'un seau générique câblé sur le nom du seul robot externe identifié historiquement (via AVA_MAGIC = 7525001). Tout nouveau robot externe hérite du même libellé fixe, à tort.
