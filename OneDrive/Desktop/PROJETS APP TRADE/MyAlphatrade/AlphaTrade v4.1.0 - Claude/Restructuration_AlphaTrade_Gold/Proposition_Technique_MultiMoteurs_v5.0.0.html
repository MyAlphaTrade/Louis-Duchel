2. Le contrat "moteur IA"

Chaque moteur est un module Python autonome qui respecte la même interface. L'AI Brain ne connaît jamais les détails internes d'un moteur — seulement ce contrat :

# Contrat commun à tous les moteurs — un module par moteur
class EngineContract:
    id: str                        # "alphatrade_ai", "kb1000_gold_ai", ...
    label: str                     # "AlphaTrade AI"
    description: str
    capabilities: dict[str, bool]  # fiche d'identité — voir §6

    def param_schema() -> dict:
        # groupes de champs, tooltips, valeurs par défaut — voir §7
        ...

    def analyze_entry(symbol, params, market_data) -> dict:
        # -> {signal, confidence, reason, details...}
        ...

    def manage_position(position, params, market_data) -> str | None:
        # -> raison de clôture, ou None si la position reste ouverte
        ...

L'entrée et la sortie appartiennent au même moteur — c'est ce qui distingue cette version de la proposition initiale de Phase 3 (qui ne rendait interchangeable que la gestion de position).

3. Le registre des moteurs — source unique de vérité

ENGINE_REGISTRY = {
    "alphatrade_ai": AlphaTradeAIEngine(),
    "kb1000_gold_ai": KB1000GoldAIEngine(),
    # futurs moteurs : une ligne de plus, rien d'autre à toucher
}

def active_engine(params: dict) -> EngineContract:
    engine_id = params.get("active_engine", "alphatrade_ai")
    return ENGINE_REGISTRY.get(engine_id, ENGINE_REGISTRY["alphatrade_ai"])

Ajouter un futur moteur (Institutional AI, London AI, News AI…) = une classe + une ligne dans ce dictionnaire. Le sélecteur Paramètres, la fiche de capacités et le routage de l'AI Brain suivent automatiquement — aucune des trois copies dupliquées trouvées lors de l'audit Phase 3 ne peut réapparaître, puisqu'il n'existe plus qu'un seul endroit où un moteur est défini.

Deux précisions suite à ta relecture :

1. Ordre du multi-timeframe corrigé. Les deux moteurs sont maintenant notés dans le même sens décroissant (grande unité → petite unité), comme KB1000. Note technique importante à ne pas perdre en codant : ce ne sont pas deux implémentations de la même méthode. multi_timeframe_context() (AlphaTrade AI, code actuel) calcule une agrégation — chaque timeframe vote indépendamment, la moyenne + majorité donne le biais final ; aucun timeframe ne filtre les autres. KB1 (KB1000 Gold AI) sera une vraie cascade hiérarchique — D1 fixe le contexte, H4 doit s'accorder avec D1, etc., chaque niveau filtrant le suivant. Les deux se ressemblent en apparence (une liste de timeframes) mais ne doivent pas être codées de la même façon.

2. Break-Even rendu visible. Il n'avait pas de ligne dédiée — il était noyé dans "TP par paliers + Break-Even réel". C'est maintenant une ligne à part : AlphaTrade AI n'a aucun break-even aujourd'hui (uniquement le trailing par seuils $ existant, voir trade_exits.jsonl) — c'est exclusivement une capacité de KB1000 Gold AI (KB8), exactement comme spécifié à l'origine (calibré broker, jamais un nombre de pips fixe).

7. Page Paramètres générée dynamiquement

Mécanisme : param_schema() du moteur actif (groupes, champs, tooltips, valeurs par défaut) est envoyé au frontend via l'IPC déjà utilisé pour charger les paramètres. renderer.js construit le DOM des sections à partir de cette réponse — il n'existe plus de HTML figé par moteur. Changer de moteur dans le sélecteur reconstruit la page ; le "Protection du compte" (§4) reste, lui, un bloc fixe non généré, identique quel que soit le moteur.

La Documentation paramètres suit la même logique. Aujourd'hui le modal "ⓘ Documentation paramètres" n'a que 4 fiches d'exemple codées en dur (confidence_min, real_lot_cap, rebond_min_signal_pct, max_position_loss) — il faudra le rendre lui aussi généré depuis param_schema() : chaque champ déclaré par un moteur (chaque nouveau champ KB1 à KB8 inclus) porte déjà, dans son schéma, sa fiche de traçabilité (utilisation, impact, valeur par défaut, codé en dur oui/non) — donc le modal se remplit automatiquement au fur et à mesure que KB1000 Gold AI est construit, sans jamais avoir à ajouter des fiches à la main. C'est la règle de traçabilité déjà en vigueur (voir Phase 1), simplement rattachée au schéma plutôt qu'à une liste séparée.
